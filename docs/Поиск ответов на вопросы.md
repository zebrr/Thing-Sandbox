# Ответы на открытые вопросы по OpenRouter API

## 1. Удаление ответа (DELETE Response Endpoint)

OpenRouter распространяет **statless**‑вариант Responses API. В разделе *Basic Usage* официальной документации подчёркивается, что при каждой операции необходимо передавать полный контекст, потому что API **не хранит историю сообщений** – «всегда включайте всю историю беседы в каждый запрос, API не сохраняет предыдущие сообщения, поэтому контекст должен храниться на стороне клиента»[[1]](https://openrouter.ai/docs/api/reference/responses/basic-usage#:~:text=). Это означает:

* Сервер OpenRouter не создаёт ни потоков, ни цепочек ответов; параметр store принудительно равен false, поэтому результат запроса не сохраняется на стороне провайдера.

* Соответственно, у OpenRouter нет смысла в DELETE/responses/{id}: удалить нечего, потому что данные не хранятся. Параметр response_id используется лишь для сопоставления наших локальных цепочек.

* В стандартном API OpenAI удаление (DELETE/v1/responses/{id}) предназначено для удаления сохранённого ответа (при store=true) и **не** очищает цепочку целиком; OpenRouter не поддерживает хранение, поэтому этот энд‑пойнт не реализован.

* Если необходимо «удалить» или обрезать цепочку, это делается на клиентской стороне: наш OpenAIAdapter оставляет метод delete_response(), но в реализации для OpenRouter он просто удаляет локальные записи (Entity._openai), не делает сетевой запрос.

## 2. Ограничения на длину цепочки (Response Chain Limits)

Поскольку OpenRouter не ведёт состояние беседы, сервер не накладывает ограничений на длину цепочки сообщений. История хранится целиком у клиента, и в каждый вызов необходимо передавать весь контекст[[1]](https://openrouter.ai/docs/api/reference/responses/basic-usage#:~:text=). Ограничения на длину цепочки возникают лишь по двум причинам:

1. **Ограничение контекстного окна модели.** Любой LLM имеет максимальное число токенов (128 к — 400 к для моделей GPT‑5.* в нашем проекте). Если суммарная длина всех предыдущих сообщений и нового запроса превышает это окно, провайдер вернёт ошибку. В обсуждении Responses API на форуме OpenAI отмечается, что при отсутствии тримминга «длина чата будет расти до предела модели, после чего запрос завершится ошибкой»[[2]](https://community.openai.com/t/introducing-the-responses-api/1140929#:~:text=There%20is%20still%20no%20cost,at%20the%20model%E2%80%99s%20maximum%20input).

2. **Локальная политика response_chain_depth.** Наша LLMClient управляет глубиной цепочки: параметр response_chain_depth в конфигурации фаз определяет, сколько предыдущих взаимодействий мы включаем в следующий запрос. Например, для фазы 2a (response_chain_depth = 2) мы прикрепляем последние два сообщения, чтобы сохранить краткосрочный контекст. Это позволяет управлять объёмом передаваемого текста и не превышать максимальный размер контекста. Если response_chain_depth = 0, запросы независимы.

Итог: OpenRouter не ограничивает глубину цепочек, но разработчик должен следить за тем, чтобы суммарное количество токенов не превышало контекстное окно конкретной модели. Для этого мы используем response_chain_depth и при необходимости выполняем сжатие/резюме.

## 3. Параметр truncation

Параметр truncation в OpenAI/Responses API определяет, как поступать, когда входящий запрос с учётом истории сообщений превышает контекстное окно модели. Официальная спецификация Responses API описывает поведение так:

* auto (наш выбор по умолчанию). Если суммарный размер входа превышает контекст, сервис **автоматически обрезает историю**, удаляя элементы с начала цепочки, пока вход не поместится в контекст[[3]](https://platform.openai.com/docs/api-reference/responses/create#:~:text=truncation). Это позволяет избежать ошибки и получить ответ, но следует помнить, что старые сообщения и даже результаты инструментов (например, tool\_call) могут быть удалены.

* disabled (умолчание для API). В этом режиме автоматическое обрезание отключено; если вход превышает контекст, API возвращает **ошибку 400**[[3]](https://platform.openai.com/docs/api-reference/responses/create#:~:text=truncation). Это поведение предпочтительнее, если потеря контекста недопустима, но тогда ответственность за управление размером истории полностью лежит на клиенте.

OpenRouter использует тот же формат запроса, что и OpenAI Responses, поэтому семантика truncation такая же. Поскольку OpenRouter не хранит историю самостоятельно, truncation применяется только к тем сообщениям, которые клиент передаёт в запросе (или к цепочке, если передан previous_response_id). В нашем проекте параметр truncation указывается в конфигурации фаз (см. config.toml) и по умолчанию стоит "auto", чтобы предотвратить ошибки при превышении контекстного окна. Однако рекомендуется контролировать размер цепочки через response_chain_depth и использовать truncation="disabled" на этапах, где потеря исторических данных недопустима (например, при генерации связных ответов).

### Вывод

* **DELETE Response**: в OpenRouter нет сервера для хранения откликов, поэтому удалять нечего; метод delete_response используется только локально.

* **Ограничение цепочек**: OpenRouter не накладывает жёстких ограничений на глубину; мы управляем её в LLMClient с учётом максимального контекста модели.

* **truncation**: параметр управляет обрезанием истории при превышении контекста. Значение auto автоматически удаляет самые старые сообщения, disabled даёт ошибку и требует ручного контроля; OpenRouter наследует эти правила от OpenAI.

Эти ответы можно включить в документ OpenRouter Responses API Reference для уточнения поведения API.
